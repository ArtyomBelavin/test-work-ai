# ERA2 Frontend — Модуль очереди генераций

Реализация интерфейса и бизнес-логики для управления очередью задач генерации (текст, изображения, видео, аудио) в рамках дизайн-системы **ERA2 (Warm Coal)**.

---

## 🛠 Архитектура и технические особенности

### 1. Маршрутизация и UI

- `/queue` — основная страница со списком задач, фильтрацией, поиском и сортировкой.
- `/chat-status` — изолированная песочница / визуальный узел для тестирования глобального плавающего индикатора состояния генераций.
- **Стилизация**: Интерфейс полностью адаптирован под темную тему `warm-coal`. Используется гарнитура `Geist` (`@fontsource-variable/geist`) и нативные CSS-токены проекта.

### 2. Управление состоянием (State Management)

- Глобальное состояние очереди инкапсулировано в контексте `QueueProvider` (`features/generation-queue/model/QueueProvider.tsx`).
- Данные синхронизированы между страницей очереди (`/queue`) и плавающим статус-баром.
- **Персистентность**: Состояние очереди автоматически сохраняется в `localStorage` (ключ `era2:generation-queue`).

### 3. Логика обработки задач (Queue Engine)

- **Многопоточность**: Симулируется параллельное выполнение задач с ограничением `MAX_CONCURRENT = 2`.
- **Диспетчеризация**: Обработка очереди происходит по принципу **FIFO** (First In, First Out). При завершении, ошибке или отмене текущего таска автоматически стартует следующий.
- **Безопасность при перезагрузке**: Задачи, имевшие статус `running` в момент закрытия вкладки, при повторной инициализации безопасно переводятся обратно в статус `queued` (в очереди). Это предотвращает зависание бесконечных таймеров на клиенте.

## 📈 Направления для рефакторинга и оптимизации (Backlog / Ревью)

В ходе реализации были зафиксированы архитектурные компромиссы и стандарты подходов к коду, вынесенные на обсуждение:

### 1. Семантическое именование компонентов (Naming Convention)

- **Контекст**: Для улучшения читаемости и явного разграничения ответственности сущностей в рамках FSD, я показал по примеру **GenerationQueue**.

### 2. Разделение кода и оптимизация Bundle Size (Code Splitting)

- **Проблема**: На данный момент страницы и тяжелые виджеты импортируются напрямую, что увеличивает размер стартового бандла и замедляет Time to Interactive (TTI).
- **Решение**: Внедрить `React.lazy` и `Suspense` (или `next/dynamic`, в зависимости от окружения) для динамической загрузки страниц и второстепенных панелей.

### 3. Декомпозиция компонентов и типов (Чистота кода)

- **Проблема**: В ряде компонентов бизнес-логика, TypeScript-интерфейсы и конфигурационные объекты находятся в одном файле с UI-разметкой (например, в `RightPanel.tsx`).
- **Решение**: Придерживаться методологии FSD. Выносить типы, утилитарные функции и селекторы в отдельные файлы внутри сегментов `model`, `lib` или `config` (как это начато в `widgets/layout/model`).

### 4. Консолидация утилит стилей (DRY)

- **Проблема**: Повторяющиеся группы классов (например, конфигурации иконок или общие стили интерактивных кнопок вроде `classNameIconBtn`) размыты по компонентам.
- **Решение**: Вынести часто используемые конфигурации Tailwind-классов в изолированный файл конфигурации (например, `текущий-сегмент/config/classNames.ts`) для переиспользования.
